# v0.1.8 Transient Multimodel Retry

## Goal

Make cold-city multimodel analysis recover automatically from short Meteoblue or Worker queue spikes instead of leaving the user with a failed analysis surface.

## Changes

- Treat transient upstream failures as retryable in the frontend multimodel loader:
  - `UPSTREAM_BAD_STATUS`
  - `UPSTREAM_FETCH_FAILED`
  - `MULTIMODEL_PAGE_TIMEOUT`
  - `MULTIMODEL_HIGHCHARTS_TIMEOUT`
  - `MULTIMODEL_INSIGHT_UNAVAILABLE`
  - `MULTIMODEL_DISTRIBUTION_UNAVAILABLE`
- Reduce multimodel queue wait from 30s to 8s so queue saturation returns quickly enough for the frontend retry loop to recover within its 20s client timeout.

## Validation

- `npm run check`
- `npm run build`
- `npm test`
- Targeted regression: `npm test -- tests/meteoblue-service.test.ts tests/app.test.ts tests/frontend-api.test.ts`

## Production Finding That Triggered This Patch

After v0.1.7, several cities that failed in the 52-city sweep succeeded on immediate retry. That means the remaining class is largely transient, so the product path should retry automatically instead of exposing a manual-refresh failure.
