# 显示契约统一与乱码防回归计划

- 时间：2026-04-03 21:51:12
- 快照：`D:\weather\tmp-snapshot-20260403215112_display-contract-hardening.zip`

## 目标

- 这轮不再做一次性的乱码补丁，而是收口前后端显示契约，避免同类问题再次进入首页和分析工作区。
- 在不改路由、不改核心交互、不破坏现有效果的前提下，修复当前用户可见乱码，并提高后续可维护性。

## 已确认根因

- 后端与前端各自维护了一份天气响应契约，存在漂移风险。
- 后端天气中文、前端 UI 中文、页面兜底文案分散在多个文件里，文案来源不单一。
- `App.tsx`、首页组件、分析组件、模型资料表、后端翻译规则表都混入了损坏的中文字面量。
- 现有编码检查只拦截极少量 mojibake token，无法覆盖真实的损坏文案样本。

## 本轮范围

- 后端：
  - 修复 `src/providers/meteoblue/week.ts` 中天气中文翻译与兜底文案的损坏文本。
  - 明确后端中文语义出口仍以 `report.textZh` 与 `hourly.items[].summaryZh` 为主。
- 前端：
  - 将 `zip/src/types.ts` 收口为基于后端共享契约的薄别名层。
  - 将首页和分析页可见中文文案集中到单一前端文案模块。
  - 清理 `App.tsx` 与核心组件中的损坏字面量，减少页面容器层重复派生。
  - 保持现有界面结构、交互和视觉基线，不做新的信息架构大改。
- 工程防线：
  - 扩展编码检测样本，覆盖当前真实乱码模式。
  - 补充针对后端中文输出与前端可见文案的回归验证。

## 验收标准

- 首页与分析工作区不再出现乱码、半中半英或损坏标点。
- 后端继续作为天气语义中文的主来源，前端只负责展示和少量状态映射。
- 前端显示契约只保留一层适配边界，不再由多个组件重复处理同一份数据。
- `npm test`
- `npm run check`
- `npm run build`

## 约束

- 不新增 endpoint，不改现有 query 语义。
- 不改首页 / 分析工作区的核心交互路径。
- 如果实施中发现必须扩展到接口 shape 变更，再单独补 `docs/decision`。
